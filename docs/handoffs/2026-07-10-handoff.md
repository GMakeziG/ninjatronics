## 3. Architectural Decisions

- **District lore must be timeless.** Descriptions describe the enduring
  concept a district represents, never its current build state
  ("not yet open," "N chambers keyed") or specific technologies/projects
  — those belong to repositories and other concrete content, not
  districts. Stated directly by the user when rejecting the first
  District Expansion plan draft; the standard: "if you replace Rust with
  Go or Kubernetes with Nomad five years from now, the description still
  makes sense."
- **Adding a domain is not purely "folder + schema + registry entry,"**
  despite `registry.ts`'s own comment. Two more spots need a real,
  mechanical (non-branching) edit every time: the `EntityType` union in
  `scripts/world-compiler/types.ts`, and `schemas/world.schema.json`'s
  `required` array + `properties`. `discover.ts`, `validate.ts`,
  `resolve.ts`, `singleton.ts`, and `assemble.ts` remain fully generic —
  confirmed by adding Experience without touching any of them. Now
  documented in `CLAUDE.md` §3 so the next domain doesn't have to
  rediscover this.
- **Domain naming convention, now explicit:** entity `type` is singular
  (`experience`, `certification`); the `content/` folder and output
  collection are the plural form (`experiences`, `certifications`) —
  except `x-singleton` domains (`profile`/`profiles`), which stay
  singular throughout.
- **Architecture and content are separable, explicit steps for a new
  domain**, not one combined milestone. Experience domain architecture
  (v0.9.0) shipped with a verified-empty content collection; content is
  scoped entirely to v0.10.0, and will only include work history the user
  explicitly identifies as public-safe — nothing invented or inferred.
- **Certifications can be linked to a district without inventing
  anything**, using a field the schema already had
  (`certification.districtId`) but that had never been populated. A
  reminder that "what schemas already support" is worth checking before
  assuming new fields are needed.
- **Manual work-history ordering is opt-in, not required.** The
  `order` field lets a specific entry override the automatic sort;
  entries that don't set it fall back to current-first/`startDate`
  descending, and an explicit `order` always outranks the automatic rule
  when the two are mixed. User's explicit design spec during v0.9.0
  planning.
- **(Established earlier, still load-bearing) Read models never modify
  compiler output; profile/identity-shaped data is never hardcoded in
  TypeScript.** Both held throughout v0.8.0 and v0.9.0 with no exceptions.

## 4. Current Architecture

```
content/*.json
     ↓
schemas
     ↓
world compiler
     ↓
data/world.json
     ↓
world.ts
     ↓
read models
     ↓
future UI / Oracle
```

- **`content/`** domains now present: `districts` (4, all real content —
  `git-forest` open, the other three locked-but-authored),
  `guardians` (4: `archivist`, `warden`, `custodian`, `scribe`),
  `repositories` (8, GitHub-sync + manual overrides), `certifications`
  (2, both linked to `floating-citadel`), `profile` (1, singleton),
  `experiences` (0 — schema/pipeline ready, no content yet). `projects`,
  `quests`, `notes`, `skills` remain empty, unchanged this session.
- **`schemas/`** — ten entity schemas now
  (`district`, `guardian`, `repository`, `project`, `quest`, `note`,
  `skill`, `certification`, `profile`, `experience`), composed into
  `schemas/world.schema.json`. Two custom keywords, both fully generic:
  `x-ref` (relationship validation) and `x-singleton` (at-most-one
  constraint, used only by `profile`).
- **World Compiler** — unchanged in structure this session
  (`discover/load/validate/resolve/singleton/assemble/report`), but two
  enumeration points (`types.ts`'s `EntityType`,
  `world.schema.json`'s top-level `required`/`properties`) were touched
  to admit the `experience` type — see §3 for why this doesn't count as
  special-casing.
- **`src/lib/world.ts`** — typed boundary, now also exports `Experience`
  and `getExperience()`.
- **Read models** — `world-graph.ts` (now warning-free — see §5),
  `git-forest.ts`, `mission-brief.ts` (now includes `workHistory`,
  currently always `[]` until v0.10.0 content lands).
- **Future UI / Oracle** — still not built; unaffected this session.

## 6. Next Recommended Milestone

**v0.10.0 — Experience content.** Populate `content/experiences/*.json`
with real, user-approved, public-safe work history, following the exact
"ask first, never invent" content process already used for Profile
(v0.7.0) and Certifications (v0.7.0). This is the natural and only
sensible next step:

1. It's the direct continuation of a domain whose architecture just
   shipped with a verified-empty content collection — the gap is
   entirely content, zero design risk.
2. It's the biggest remaining gap in Mission Brief's stated purpose (a
   recruiter-facing summary): `workHistory` currently renders as `[]` no
   matter who consumes it.
3. It re-proves nothing new about the pipeline — the schema, compiler
   wiring, and sort logic are already verified — so risk is close to
   zero and effort is proportionate to how much work history the user
   wants to share.
4. It was already the user's explicit choice, stated at the end of the
   v0.9.0 work: "we will start a separate v0.10.0 milestone to populate
   public-safe work history content."

Lower-priority items from the original project roadmap remain unchanged
and are not re-prioritized here: a Learning Roadmap domain (planned/
in-progress certifications, kept separate from earned `certifications`),
a minimal public frontend consuming `src/lib/` read models, and
eventually a real test framework as more domains accumulate.
