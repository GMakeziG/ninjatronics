# Ninjatronics — Handoff: 2026-07-09

## 1. Executive Summary

Today's session took Ninjatronics from "a validated content pipeline with
one hand-authored repository" to a working, multi-layer read model over
real professional data — the first version of the world that a future UI
or the Oracle could actually query for something a recruiter would care
about.

**What we accomplished:**
- Built a GitHub ingestion service that pulls real public repositories
  into the compiler pipeline.
- Built three successive derived read-model layers on top of the
  compiled world: a district graph, a "repositories as trees" view, and a
  recruiter-facing Mission Brief summary.
- Introduced the project's first genuinely new compiler-level mechanism
  (a generic `x-singleton` schema keyword) to support a first-class
  Profile domain — the first change all day that touched the compiler
  itself rather than sitting downstream of it.
- Populated the world with real, user-approved profile and certification
  data, end to end, with zero hardcoded personal data anywhere in
  TypeScript.

**Architectural milestones completed:** the read-model layer now has a
proven, repeatable shape (derive → compute-once constant → thin getters,
never write back), and the compiler has proven it can grow a new generic
mechanism (`x-singleton`) without any per-type special-casing, exactly as
`HANDOFF.md`'s founding design principles demanded it should be able to.

**Current project version:** `v0.7.0` (git-tagged `v0.1.0` through
`v0.7.0`, all seven released this session). Working tree is clean; every
version described below is committed.

## 2. Completed Milestones

**GitHub repository ingestion (v0.2.0)** — `scripts/sync-github.ts` +
`scripts/github-sync/*.ts` fetch GMakeziG's public, non-fork repositories
from the GitHub API, normalize them into repository-entity defaults, and
merge in hand-authored overrides from `content-overrides/repositories/*.json`
(a directory deliberately outside `content/`, so the compiler's
`discover.ts` never walks it). Manual metadata always wins per-key over
GitHub defaults. Output is exclusively `content/repositories/*.json` —
the World Compiler remains the only thing that writes `data/world.json`.
Orphaned files are reported, never auto-deleted.

**World Graph (v0.3.0)** — `src/lib/world-graph.ts` computes a derived,
read-only district graph (`DistrictNode` per district, embedding its
guardian and repositories; directed `connections` edges) directly from the
already-validated `World` shape. Never baked into `data/world.json` — it's
a runtime projection, not new compiler output. Includes non-fatal
consistency warnings (asymmetric connections, self-loops, unreachable open
districts) and a BFS-based `getRecommendedPath()`.

**Git Forest read model (v0.4.0)** — `src/lib/git-forest.ts` reshapes
every repository in the world into a "tree" (`RepositoryTree`), scoped to
*all* repositories regardless of district (not hardcoded to the
`git-forest` district id), carrying its owning district/guardian context
straight from `world-graph.ts`'s already-computed grouping. Introduced a
mechanical `richness` score (story + technology count) to power
`getFeaturedTrees()`, explicitly documented as a placeholder proxy, not
authored curation data.

**Mission Brief read model (v0.5.0)** — `src/lib/mission-brief.ts`, the
first module that summarizes *across* the whole world (skills, learning
paths/quests, featured repositories, projects, certifications, a derived
GitHub URL) into one recruiter-facing snapshot. Shipped with `profile`/
`contact` as genuine empty placeholders — no fabricated identity data —
pending a real content source.

**Profile domain (v0.6.0)** — the first change of the whole project that
touched the compiler pipeline itself. Added `schemas/entities/
profile.schema.json`, a registry entry, and a new generic compiler
mechanism: the `"x-singleton": true` schema keyword, enforced by the new
`scripts/world-compiler/singleton.ts` module, so at most one `profile`
entity can ever validate successfully — read exactly like the existing
`x-ref` keyword, with zero per-type branching anywhere in the compiler.
`world.ts` gained `getProfile()`; `mission-brief.ts` was rewired to read
from it instead of hardcoded `{}`.

**Real profile content (v0.7.0)** — `content/profile/profile.json`:
name, professional title, tagline, location, and a GitHub link, all
explicitly provided/approved. Email, LinkedIn, and website were
deliberately left out (deferred by the user), not filled with placeholders.

**Certification content (v0.7.0)** — `content/certifications/
comptia-security-plus.json` and `azure-administrator-associate.json`:
exactly the two *earned* certifications provided. No planned/in-progress
certifications (GH-900, CCNA, LPIC-1) were written anywhere, per explicit
instruction.

## 3. Architectural Decisions

- **Repositories are trees.** Every repository entity is presented as a
  "tree" (`RepositoryTree`) regardless of which district currently owns
  it — CLAUDE.md's World Rules state this unconditionally, and nothing in
  `repository.schema.json` restricts `districtId` to one specific
  district. `git-forest.ts` flatMaps every district node's repositories
  rather than hardcoding the `git-forest` id or sniffing for a
  GitHub-`dataSource` district.
- **Read models never modify compiler output.** `world-graph.ts`,
  `git-forest.ts`, and `mission-brief.ts` are all pure, computed-once
  projections over the already-validated `World` shape. None of them
  write to `content/`, `data/world.json`, or any schema. The World
  Compiler (`scripts/compile-world.ts` + `scripts/world-compiler/`)
  remains the sole producer of `data/world.json`.
- **Profile data comes from `content/profile/profile.json`, never
  hardcoded TypeScript.** Stated and re-confirmed by the user multiple
  times across v0.5.0–v0.7.0. `mission-brief.ts`'s `profile`/`contact`
  fields are a direct mapping from `getProfile()`'s return value, with
  `{}`/empty-array fallbacks when no profile exists — never a fabricated
  default.
- **Certifications represent earned certifications only.** No
  planned/in-progress certification is ever written to
  `content/certifications/`, even as a "coming soon" entry.
- **Planned certifications belong in a future Learning Roadmap domain.**
  Explicitly deferred, not designed this session — tracked so aspirational
  content never gets mixed into the certifications collection later "for
  convenience."
- **Professional work experience will become its own future Experience
  domain, not `project.schema.json`.** `project.schema.json` models
  software projects (status/description/technologies/links) and has no
  employer/role/date-range fields. Rather than stretch it or add ad hoc
  fields mid-task, real work history is deferred to a properly-designed
  future domain (employer, role, dates, summary, highlights).
- **Stable IDs should remain independent from display names.** E.g. the
  profile entity's `id` is `"gerso"`, not a slug of
  `"Gerso Robayo-Guillen"` — short and stable beats descriptive, since ids
  are referenced internally and don't need to mirror what's displayed.
- **(Established earlier, still load-bearing today) New domains require
  only a schema + one registry line, never special-cased compiler code.**
  The `x-singleton` mechanism built for the Profile domain is the clearest
  proof point yet: a real, previously-unsolved problem (cardinality) was
  solved generically, exactly the way `CLAUDE.md` and `HANDOFF.md` both
  insist new domains must be solved.

## 4. Current Architecture

```
content/*.json
     ↓
schemas
     ↓
world compiler
     ↓
data/world.json
     ↓
world.ts
     ↓
read models
     ↓
future UI / Oracle
```

- **`content/*.json`** — hand-authored (and now, for repositories,
  ingestion-authored) source-of-truth entities, one file per entity, one
  folder per domain (`districts`, `guardians`, `repositories`, `projects`,
  `quests`, `notes`, `skills`, `certifications`, `profile`). Nothing reads
  these directly except the compiler.
- **`schemas/`** — one JSON Schema per entity type
  (`schemas/entities/*.schema.json`) plus the composed
  `schemas/world.schema.json`. Every entity is `additionalProperties:
  false` and validated before anything downstream trusts it. Two custom
  keywords exist: `x-ref` (property-level — "this field's value must be a
  real id of entity type X") and `x-singleton` (schema-root-level — "at
  most one entity of this type may exist"), both read generically by the
  compiler with zero per-type code.
- **World Compiler** (`scripts/compile-world.ts` +
  `scripts/world-compiler/{discover,load,registry,validate,resolve,
  singleton,assemble,report}.ts`) — discovers every `content/**/*.json`
  file, parses it (JSONC-tolerant, strips a leading breadcrumb comment),
  validates it against its type's schema, resolves every `x-ref` and
  checks every `x-singleton` constraint, assembles the validated entities
  into collections (sorted by id, one array per registered domain), and
  writes the result. Fails closed on any schema violation, duplicate id,
  unresolved reference, singleton violation, or unregistered folder —
  never degrades silently. This is the only thing allowed to write
  `data/world.json`.
- **`data/world.json`** — the single compiled build artifact. Never
  hand-edited; regenerated by `npm run compile:world`.
- **`src/lib/world.ts`** — typed mirror of the compiled world (`World`,
  `District`, `Guardian`, `Repository`, `Profile`, etc. interfaces) plus
  basic indexed getters (`getDistrict`, `getProfile`, ...). This is the
  single data boundary every downstream consumer is expected to import
  from directly.
- **Read models** (`src/lib/world-graph.ts`, `src/lib/git-forest.ts`,
  `src/lib/mission-brief.ts`) — pure, computed-once derived projections
  over `world.ts`. Each follows the same shape: a private `build*()`
  function, a module-level constant computed once at import time, and
  thin exported getters. None of them are re-exported back into
  `world.ts` (would risk a circular-import TDZ failure since they need
  runtime values, not just types, from it) — each sits one layer above
  the core data boundary, imported directly by whatever needs it.
- **Future UI / Oracle** — not built yet. Both are expected to consume
  `src/lib/world.ts` and the read models above as their entire data
  layer, per `HANDOFF.md`'s original design intent.

## 6. Next Recommended Milestones

1. **Expand the three locked districts** (`dojo-gate`, `floating-citadel`,
   `hall-of-knowledge`) with real subtitles, descriptions, guardians, and
   `dataSources` where relevant. This is the highest-leverage content gap:
   `world-graph.ts`'s reciprocity/reachability warnings, `getRecommendedPath`,
   and the whole "explorable world" metaphor all stay inert with only one
   real district. Low architectural risk — pure content authoring against
   schemas that already exist.
2. **Design and build the Experience domain.** This is the biggest
   remaining gap between "recruiter-friendly summary" (the stated purpose
   of Mission Brief) and reality — a résumé without work history is
   incomplete. It's also the natural next test of whether the
   schema → registry → compiler → read-model pattern generalizes cleanly a
   third time (after Profile).
3. **Stand up a minimal frontend** consuming `src/lib/world.ts` +
   `mission-brief.ts`, even a bare-bones one — every read model built so
   far has only ever been verified via disposable scratchpad scripts.
   Getting *something* rendering (even just the Mission Brief as plain
   HTML) would be the first true end-to-end proof of the whole pipeline
   and would surface any read-model shape issues before more layers are
   built on top.
4. **Learning Roadmap domain** — lower priority than Experience, since
   there's no user-facing consumer waiting on it yet, but cheap to build
   once Experience has re-proven the domain-creation pattern.

Priority order reasoning: content gaps (#1) are safest and highest-value
per effort; Experience (#2) closes the single biggest gap in the stated
"recruiter résumé" goal; a minimal frontend (#3) is the first real
end-to-end validation of everything built so far and de-risks all future
read-model work; Learning Roadmap (#4) has no urgency until Experience
proves the pattern again.

  minimum the Mission Brief ("press B") view and the Git Forest, backed
  entirely by the existing `src/lib/` read-model layer with no new data
  modeling required. This is the natural "v1" line — everything before it
  is world-model plumbing; v1.0.0 is the first version an actual visitor
  could look at.

## 8. Open Questions

- **Should `content/profile/profile.json` ever be allowed to have more
  than the one file it has today** (e.g. split hand-authored vs.
  sync-generated pieces, mirroring `content-overrides/repositories/`), or
  is a single hand-edited file sufficient indefinitely? Not urgent — no
  upstream automation exists yet.
- **Should the three locked districts get real `dataSources`** (e.g.
  Obsidian for `hall-of-knowledge`) as part of District Expansion, or is
  that a separate integration milestone of its own?
- **Should `git-forest.ts`'s `richness` heuristic eventually be replaced
  by an explicit, authored "featured" flag**, once there's enough real
  repository content to make the mechanical proxy feel wrong? No decision
  needed yet — flagged in code as a placeholder, revisit when it starts
  producing a featured list the user disagrees with.
- **Where should Experience-domain content actually live relative to
  Projects** — should a `project` ever reference an `experience` (e.g.
  "built while working at X"), or should the two domains stay fully
  decoupled? Needs a decision before v0.9.0 schema design starts.
- **Is a test framework worth introducing before or after the Experience
  domain?** All verification to date has been manual/scratchpad-based;
  this has worked so far but doesn't scale indefinitely as more read
  models and domains accumulate.

## 9. Tomorrow's Starting Point

**Start here tomorrow**: begin **v0.8.0 — District Expansion**
(`content/districts/dojo-gate.json`, `floating-citadel.json`,
`hall-of-knowledge.json`). Reasons to start here specifically, over the
other options in the roadmap:

1. It's pure content work against schemas that already exist — zero
   compiler or schema risk, the safest possible next step.
2. It directly resolves real, currently-active warnings:
   `world-graph.ts`'s `buildWorldGraph()` today reports 3 live
   `asymmetric-connection` warnings (the 3 stub districts don't connect
   back to `git-forest`) purely because they have no real content yet —
   this milestone is the natural fix, not a new mechanism.
3. It unblocks meaningful testing of `getRecommendedPath()` and the
   reachability warning logic, which have only ever been exercised
   against a degenerate one-real-district graph so far.
4. It's a natural warm-up before the more architecturally significant
   Experience domain (v0.9.0) — same "schema-driven content authoring"
   muscle, lower stakes.

Before writing any district content, re-read `content/districts/
git-forest.json` as the one fully-authored example, and ask the user for
the same kind of explicit, approved data this session used for Profile
(subtitle, description, guardian assignment, connections) rather than
inventing district lore unilaterally — consistent with every content
decision made today.
