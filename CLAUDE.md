# CLAUDE.md — Ninjatronics Operating Guide

This file is the permanent operating guide for Claude Code sessions
working on Ninjatronics. Read it before making changes.

## 1. Project Vision

Ninjatronics is not a normal portfolio. It is a living digital world, a
second brain, and a cyber dojo. GitHub, Obsidian, AI, projects, quests,
certifications, and skills will eventually feed one evolving world rather
than existing as separate disconnected pages.

## 2. Core Architecture

- Source content lives under `content/`, one folder per domain (districts,
  guardians, repositories, projects, quests, notes, skills,
  certifications).
- `data/world.json` is **compiled output**. Do not hand-edit
  `data/world.json` unless explicitly asked — it is regenerated by the
  World Compiler and any manual edit will be silently overwritten on the
  next build.
- The World Compiler (`scripts/compile-world.ts` and
  `scripts/world-compiler/`) discovers, validates, resolves, and assembles
  `content/**/*.json` into `data/world.json`.
- Schemas live under `schemas/` (`schemas/entities/*.json` per type, plus
  the composed `schemas/world.schema.json`).
- Typed helpers for consuming the compiled world live under
  `src/lib/world.ts`.

## 3. Development Rules

- Preserve the schema-driven architecture. Content is validated against a
  schema before it is trusted anywhere downstream.
- Add new domains through a schema + registry entry
  (`scripts/world-compiler/registry.ts`), not by special-casing code in
  the compiler pipeline.
- Use IDs for relationships between entities — never embed a copy of one
  entity's data inside another.
- Fail closed when references are missing. A broken reference, schema
  violation, duplicate ID, or unregistered content folder should stop the
  build with a clear error, not degrade silently.
- Avoid hard-coded relationships. Which fields reference which entity
  types is schema metadata (`x-ref`), read generically — not an `if`
  statement per entity type.
- Keep modules small and focused. Each stage of the compiler
  (discover/load/validate/resolve/assemble/report) does one job.

## 4. Safety Rules

- Explain the plan before major file moves or Git operations.
- Ask before installing packages.
- Ask before changing permissions.
- Ask before deleting files.
- Ask before running `git commit`, `git push`, `git reset`, or
  `git rebase`.
- Never touch secrets or `.env` files without explicit approval.

## 5. Coding Standards

- Use TypeScript.
- Prefer clear, readable code over cleverness.
- Validate JSON content — don't trust `content/**/*.json` without running
  it through the schema-validated compiler pipeline.
- Keep compiler logic modular; resist collapsing pipeline stages together.
- Use the npm scripts already defined (`npm run compile:world`,
  `npm run compile:world:check`) unless explicitly asked to change package
  managers or scripts.

## 6. Git Standards

Use Conventional Commits. Examples:

- `feat: add Git Forest district`
- `fix: resolve guardian reference validation`
- `refactor: simplify world compiler registry`
- `docs: update project handoff`

## 7. Ninjatronics World Rules

- Districts are world zones.
- Guardians represent disciplines.
- Repositories become trees.
- Notes become knowledge artifacts.
- Certifications unlock chambers.
- Quests represent learning paths.
- The Oracle reads from compiled world knowledge.

4. Frontend/UI comes after the world data model is stable.
