# Ninjatronics — Development Handoff

## Project Vision

Ninjatronics is not a portfolio website. It is intended to become a living
digital world and second brain — a persistent, explorable space that grows
as its creator learns, ships, and studies.

Over time, the world should integrate:

- **GitHub** — repositories become trees, commits become rings, releases
  become lanterns
- **Obsidian** — notes become books in the Hall of Knowledge
- **AI (the Oracle)** — a knowledge layer that answers questions from the
  compiled world itself
- **Projects** — explorable buildings tied to real work
- **Certifications** — unlockable chambers/districts
- **Quests** — structured learning paths with objectives and rewards
- **Weather and time** — ambient systems that reflect real activity

The unifying idea: the world is a *representation* of real work and
learning, not a hand-authored fiction layered on top of it. Every system
built from here forward should serve that metaphor.

## Current Architecture

**Schema-driven content.** Every entity (district, guardian, repository,
project, quest, note, skill, certification) is defined by its own JSON
Schema under `schemas/entities/`. Content files under `content/` are
validated against these schemas — nothing is assembled until it's valid.

**World Compiler.** `scripts/compile-world.ts` orchestrates a small
pipeline in `scripts/world-compiler/`:

```
discover -> load (JSONC) -> validate (Ajv) -> resolve (ID refs) -> assemble -> write
```

- `discover.ts` — recursively finds every `.json` file under `content/`
- `load.ts` — parses JSON, tolerating a leading `// path/to/file.json`
  comment line used as a human-readable breadcrumb
- `validate.ts` — loads all entity schemas plus the composed
  `schemas/world.schema.json` into one Ajv instance
- `resolve.ts` — checks that every ID reference points at something real
- `assemble.ts` — groups validated entities into their output collections
- `report.ts` — prints grouped, per-file, colorized error output

**Registry system.** `scripts/world-compiler/registry.ts` is the single
place that maps a `content/` folder to its entity type, output collection
name, and schema file. It is the only file that needs to change to add a
brand-new domain to the world.

**Entity validation.** Each schema is strict (`additionalProperties:
false`) so typos and unexpected fields are caught immediately rather than
silently passed through to the compiled output.

**Cross-reference resolution.** Reference fields are declared in-schema
with a custom `"x-ref": "<entityType>"` keyword (on the property itself for
single references, or on `items` for arrays — e.g. a district's
`connections` list). The compiler reads this metadata generically at
runtime; it does not hard-code which fields reference which types anywhere
in code.

**Compiled `data/world.json`.** The single build artifact the frontend
(and eventually the Oracle) should consume. It is never hand-edited — it is
regenerated by `npm run compile:world`.

**Typed interfaces in `src/lib/world.ts`.** A hand-written TypeScript
mirror of the compiled shape (`District`, `Guardian`, `Repository`,
`Project`, `Quest`, `Note`, `Skill`, `Certification`, `World`), plus
indexed getters (`getDistrict`, `getGuardianForDistrict`,
`getConnectedDistricts`, etc.) so consumers never hand-roll a `.find()`
over the raw JSON.

### Why this architecture, and why it must stay modular

The whole point of splitting content into per-domain JSON files instead of
one giant `world.json` is that this project's own stated future work —
GitHub sync, Obsidian import, the Oracle — are all just *additional
producers of `content/**/*.json`*. None of them should ever need to know
about each other, and none of them should write directly to
`data/world.json`. The compiler is the only thing allowed to produce that
file, and it does so generically, driven by schema + registry metadata
rather than per-type logic. As long as that boundary holds, new
integrations are additive: a new content folder, a new schema, one
registry line — never a change to `discover.ts`, `resolve.ts`, or
`assemble.ts` themselves.

## Design Principles

- **IDs are the source of truth.** Entities reference each other by ID
  string, never by embedding a copy of another entity's data.
- **Every content type lives in its own folder.** One domain, one folder
  under `content/`, one schema, one registry entry.
- **The World Compiler assembles the world.** Nothing else is allowed to
  write `data/world.json`. It is a build artifact, not a source file.
- **Fail closed when references are invalid.** A broken reference, a
  schema violation, a duplicate ID, or an unregistered content folder all
  stop the build and print a clear per-file error — the compiler never
  silently drops or guesses.
- **Never hard-code relationships.** Which fields reference which entity
  types is schema metadata (`x-ref`), read generically by the resolver —
  not an `if` statement per entity type in compiler code.
- **New domains should require only a schema and a registry entry.** If
  adding GitHub sync, Obsidian import, or a new gameplay system ever
  requires touching `discover.ts`, `resolve.ts`, or `assemble.ts`, that's a
  signal the abstraction has leaked and should be reconsidered.

## Technical Debt

- **No frontend yet.** `src/lib/world.ts` exists as a typed data layer, but
  there is no framework (Next.js/Astro/Vite/etc.) wired up to actually
  render the world. This was intentionally deferred — no framework choice
  has been made.
- **Content folders are mostly empty stubs.** `projects/`, `quests/`,
  `notes/`, `skills/`, `certifications/` have schemas but zero content.
  Their schemas are best-effort/forward-looking and may need adjustment
  once real content is authored against them.
- **Stub districts are placeholders.** `dojo-gate`, `floating-citadel`, and
  `hall-of-knowledge` only have `id`/`name`/`slug`/`status: "locked"` — no
  guardian, description, or connections. They exist solely to satisfy
  reference validation and need real content.
- **No automated test suite.** The compiler was verified manually (failing
  case, then passing case, plus a `tsc --noEmit` type-check). There are no
  unit tests for `discover`/`load`/`resolve`/`assemble` in isolation yet.
- **No CI.** Nothing runs `compile:world:check` automatically on push/PR
  yet — that's a natural next step once GitHub integration begins.
- **`world.schema.json` validation is a redundant-but-cheap safety net.**
  Every entity is already validated individually before assembly, so the
  final whole-world validation pass should always pass if the per-entity
  passes did. Kept as a sanity check in case assembly logic ever
  desyncs from the schemas.
- **No `lint` script actually configured.** The permission trust model
  pre-allows `npm run lint`, but no linter (ESLint/etc.) is installed yet.

## Future Vision

The long-term goal is for the world itself to evolve from real activity,
not from manual edits:

- GitHub commits grow trees
- Certifications unlock districts
- Notes become books
- Skills level up automatically from real usage/evidence
- The Oracle answers questions from world knowledge, not a static FAQ
- Weather reflects activity (a quiet week feels different from a shipping
  week)
- Projects become explorable buildings, not list items

Every one of these is, mechanically, "some external signal produces or
updates a `content/**/*.json` file, and the World Compiler does the rest."
That's the reason the compiler was built as a generic, schema-driven
pipeline instead of a one-off script — it needs to be the stable center
that all of these very different data sources can safely write into.

## Development Philosophy

Favor maintainability, extensibility, and storytelling over shortcuts.
This project has no external deadline pressure forcing shortcuts — take
the time to keep the compiler generic, the schemas honest, and the content
model clean.

Every major feature should strengthen the world metaphor rather than
breaking immersion. Before adding a field, a page, or a system, ask
whether it reads as part of the world (a tree, a guardian, a chamber) or
as raw admin/CRUD leaking through. When in doubt, favor the metaphor.

## Session Notes

- The repo's real Git history was initially nested one directory below the
  working root (`~/src/ninjatronics/ninjatronics`), separate from where
  the new compiler work was being built. This was discovered early by
  inspecting `git remote -v` and `git log` rather than assuming — worth
  doing that check early in any session that touches repo structure.
- Content files intentionally use a leading `// content/path/file.json`
  comment line. This is not valid strict JSON; the loader strips only
  whole comment lines (lines whose first non-whitespace characters are
  `//`) rather than doing a naive regex strip, so it won't corrupt strings
  that happen to contain `//` (e.g. URLs).
- The compiler was deliberately exercised in a *failing* state first
  (`git-forest.json` referencing three nonexistent districts) before being
  fixed — this proved the fail-closed error reporting actually works,
  rather than only ever being seen on a happy path.
- Before any bulk file move/flatten, every directory was diffed
  (`diff -rq`) to confirm true duplication before deleting anything. No
  destructive operation was run without that verification step first.
- A permission trust model now lives in `.claude/settings.json`: reads,
  listing, lint, typecheck, build, and test are auto-approved; file edits,
  package installs, and git writes require review.
